Ringkasan Strategi Testing

Apa Itu Testing dan Tujuannya?

Testing (Pengujian) adalah proses mengevaluasi produk perangkat lunak untuk menemukan cacat (bug) dan memastikan produk bekerja sesuai kebutuhan, baik fungsional maupun non-fungsional.

Tujuan Utama Testing:

  • Menemukan kesalahan dan memastikan kualitas produk sebelum rilis.
  • Melakukan Verifikasi dan Validasi.
  • Mengurangi risiko kegagalan sistem.
  • Menjamin keamanan dan efisiensi biaya.
  • Meningkatkan kepercayaan stakeholder dan pengalaman pengguna.

Software Testing Life Cycle (STLC)

STLC adalah pendekatan sistematis untuk menguji perangkat lunak. Berikut adalah tahapannya:

  1. Test Planning (Perencanaan): Membuat strategi, mengidentifikasi lingkungan, dan memperkirakan waktu/biaya.
  2. Test Design (Perancangan): Menulis kasus uji (test case), membuat data pengujian, dan menentukan hasil yang diharapkan.
  3. Test Execution (Eksekusi): Menjalankan pengujian pada berbagai level (Component, Integration, System, Acceptance).
  4. Pelaporan & Analisis: Meringkas hasil, mencatat bug, dan memberi rekomendasi perbaikan.

Klasifikasi Software Testing

Berdasarkan Abstraksi (Level)

  • Unit Testing: Menguji komponen terkecil.
  • Integration Testing: Menguji interaksi antar unit.
  • System Testing: Menguji sistem secara keseluruhan.
  • Acceptance Testing: Menguji dari perspektif klien.

Berdasarkan Fungsi

  • Functional Testing: Menguji apakah fungsi berjalan (Misal: Tombol login).
  • Non-Functional Testing: Menguji bagaimana sistem bekerja (Misal: Kecepatan).

Berdasarkan Domain

  • Performance Testing: Menguji kecepatan & stabilitas.
  • Security Testing: Mengidentifikasi celah keamanan.
  • Usability Testing: Mengevaluasi kemudahan penggunaan.

Berdasarkan Struktur (Metode)

  • Black-Box Testing: Tester tidak tahu kode (fokus di input/output).
  • White-Box Testing: Tester tahu kode (fokus di alur logika).

Kesimpulan

Testing adalah tahapan krusial dalam Software Development Life Cycle (SDLC) untuk menghasilkan perangkat lunak yang berkualitas, bebas bug, dan disukai pengguna.

Ringkasan Materi UI/UX Testing

1. Perbedaan Mendasar UI vs. UX Testing

UI (User Interface) Testing

Berfokus pada tampilan visual dan elemen antarmuka. Tujuannya memastikan warna, ikon, ukuran tombol, dan tata letak (layout) ditampilkan dengan benar, konsisten, dan responsif.

UX (User Experience) Testing

Berfokus pada keseluruhan pengalaman pengguna. Tujuannya menguji apakah pengguna dapat mencapai tujuan mereka (misal: menyelesaikan pembelian) dengan mudah, efisien, dan tanpa kebingungan.

2. Fokus Utama Pengujian

Fokus Utama UI Testing

  • Konsistensi Visual: Memastikan gaya seragam (warna, ikon, font) di semua halaman.
  • Responsivitas: Memeriksa tampilan dan fungsionalitas di berbagai ukuran layar (desktop, tablet, mobile).
  • Kompatibilitas: Memastikan UI bekerja baik di berbagai browser (Chrome, Firefox, Safari) dan sistem operasi (Windows, macOS, Android, iOS).

Fokus Utama UX Testing

  • Kegunaan (Usability): Seberapa mudah dan efektif pengguna berinteraksi untuk menyelesaikan tugas.
  • Aksesibilitas (Accessibility): Memastikan software dapat digunakan oleh semua orang, termasuk penyandang disabilitas (misal: pengujian *screen reader* & kontras warna).
  • Alur Pengguna (User Flow): Menganalisis kelancaran pengguna berpindah dari satu langkah ke langkah berikutnya (misal: dari *homepage* ke *checkout*).

3. Metode Pengujian Umum

A/B Testing

Metode perbandingan dua versi (A dan B) dari sebuah halaman untuk melihat mana yang lebih efektif dalam mencapai tujuan (misal: lebih banyak klik tombol).

Contoh Tools: Google Optimize, VWO, Optimizely.

Heatmaps

Visualisasi data yang menunjukkan di mana pengguna paling sering mengklik, menggerakkan mouse (*move map*), atau men-scroll halaman (*scroll map*).

Contoh Tools: Hotjar, Crazy Egg, Microsoft Clarity.

Usability Testing

Mengamati pengguna nyata saat mereka mencoba menyelesaikan tugas. Tujuannya adalah menemukan masalah alur, kebingungan, dan hambatan secara langsung.

Contoh Tools: Maze, UserTesting.com, Lookback.

4. Evaluasi Heuristik (10 Prinsip Desain Nielsen)

  1. Visibilitas Status Sistem: Sistem harus memberi tahu pengguna apa yang sedang terjadi (umpan balik).
  2. Kecocokan Sistem & Dunia Nyata: Gunakan bahasa dan konsep yang familiar bagi pengguna.
  3. Kontrol dan Kebebasan Pengguna: Sediakan "pintu darurat" untuk membatalkan aksi.
  4. Konsistensi dan Standar: Elemen desain dan terminologi harus konsisten.
  5. Pencegahan Kesalahan: Desain yang baik harus mencegah pengguna melakukan kesalahan.
  6. Mengenali Daripada Mengingat: Minimalkan beban ingatan pengguna (buat objek/pilihan terlihat).
  7. Fleksibilitas dan Efisiensi: Sediakan *shortcuts* untuk pengguna ahli.
  8. Desain Estetis dan Minimalis: Antarmuka tidak boleh berisi info yang tidak relevan.
  9. Membantu Pemulihan Kesalahan: Pesan eror harus mudah dimengerti dan menyarankan solusi.
  10. Bantuan dan Dokumentasi: Bantuan harus mudah dicari dan fokus pada tugas pengguna.

Ringkasan Materi: Testing Plan

1. Apa itu Testing Plan?

Testing Plan (Rencana Pengujian) adalah dokumen panduan strategis yang menjelaskan bagaimana proses pengujian perangkat lunak akan dilakukan. Dokumen ini berfungsi sebagai acuan resmi bagi tim penguji agar prosesnya terarah dan konsisten.

Tujuan Utama:

  • Memberi gambaran jelas tentang apa yang akan diuji dan bagaimana cara mengujinya.
  • Memastikan proses pengujian menemukan sebanyak mungkin kesalahan (bugs).
  • Menjamin perangkat lunak mencapai kualitas yang diharapkan pengguna.
  • Mengoptimalkan penggunaan waktu, biaya, dan sumber daya.
  • Menyediakan dokumentasi untuk evaluasi proyek di masa depan.

2. Komponen Kunci Testing Plan

Dokumen Test Plan memiliki banyak komponen (seperti standar IEEE 829). Berikut adalah pengelompokan komponen-komponen terpenting dari materi tersebut:

Komponen: Ruang Lingkup (Scope)

  • Test Items: Komponen atau modul teknis yang akan diuji.
  • Features to be Tested: Fitur yang akan diuji dari sudut pandang pengguna.
  • Features not to be Tested: Fitur yang sengaja dikecualikan dari pengujian, beserta alasannya (misal: sudah stabil, tidak dirilis).
  • Software Risk Issues: Identifikasi potensi risiko (misal: fitur kompleks, integrasi baru, kebutuhan yang samar).

Komponen: Pendekatan (Approach)

Menentukan strategi umum pengujian, mencakup:

  • Tipe Pengujian: (Unit, Integration, System, Acceptance).
  • Teknik Pengujian: (Black-box, White-box).
  • Metode Pengujian: (Manual atau Otomatis).
  • Tujuan Pengujian: (Validasi fungsionalitas, kinerja, keamanan).

Komponen: Kriteria & Aturan

  • Item Pass/Fail Criteria: Standar yang jelas dan objektif untuk menentukan apakah sebuah fitur lulus atau gagal uji.
  • Suspension Criteria: Kondisi kapan pengujian harus dihentikan sementara (misal: ditemukan bug kritis yang memblokir).
  • Resumption Requirements: Kondisi yang harus dipenuhi agar pengujian dapat dilanjutkan kembali.

Komponen: Logistik, Tim, & Jadwal

  • Environmental Needs: Kebutuhan lingkungan pengujian (Hardware, Software/Versi, Data Uji, Alat, Jaringan).
  • Responsibilities: Pembagian tanggung jawab (Siapa menguji apa, siapa memverifikasi bug).
  • Staffing & Training Needs: Penentuan peran (Test Manager, Tester) dan kebutuhan pelatihan tim.
  • Schedule: Garis waktu (*timeline*) pengujian, termasuk *milestone* penting dan jadwal *retest*.

3. Dokumentasi & Hasil Akhir

Test Deliverables (Hasil)

Dokumen, artefak, dan hasil nyata yang diserahkan, seperti: Laporan Bug, Hasil Eksekusi Test Case, dan Laporan Ringkasan Pengujian (*Test Summary Report*).

Approvals (Persetujuan)

Bagian akhir yang berisi tanda tangan dari *stakeholder* (seperti Manajer Proyek) yang menyetujui ruang lingkup, jadwal, dan sumber daya dalam rencana ini.

Test Scenario, Test Case, & Bug Report

1. Alur Kerja Dokumen Pengujian

Test Scenario

"Apa yang akan diuji?"
(Gambaran Umum)

Test Case

"Bagaimana cara mengujinya?"
(Langkah-langkah Detail)

Bug Report

"Apa hasil temuannya?"
(Jika Hasil = Gagal)

2. Anatomi Dokumen

Anatomi Test Scenario & Test Case

Test Scenario (Level Tinggi)

  • ID Scenario: Kode unik (misal: TS001).
  • Modul/Fitur: Fitur yang diuji (misal: "Kalkulator BMI").
  • Deskripsi: Penjelasan singkat *apa* yang diuji (misal: "Verifikasi perhitungan dan klasifikasi BMI").

Test Case (Level Detail)

  • ID Test Case: Kode unik (misal: TC001).
  • Precondition: Kondisi awal (misal: "Aplikasi terbuka di halaman utama").
  • Test Steps: Langkah-langkah detail (1. Masukkan tinggi 170cm, 2. Masukkan berat 60kg, 3. ...).
  • Test Data: Input yang digunakan (Tinggi = 170, Berat = 60).
  • Expected Result: Hasil yang diharapkan (misal: "Hasil BMI = 20.8 dan kategori 'Normal'").
  • Actual Result: Hasil yang sebenarnya terjadi (diisi oleh tester).
  • Status: Lulus / Gagal (Pass / Fail).

Anatomi Bug Report

  • Bug ID: Kode unik bug (misal: BMI-001).
  • Bug Title: Judul jelas & ringkas (misal: "Perhitungan BMI salah saat input 170cm & 60kg").
  • Step to Reproduce: Langkah pasti untuk memicu bug (Paling Penting!).
  • Actual Result: Perilaku salah yang terjadi (misal: "Hasil BMI = 12.5").
  • Expected Result: Perilaku yang seharusnya terjadi (misal: "Hasil BMI seharusnya = 20.8").
  • Severity: Seberapa besar *dampak* bug ini? (Lihat di bawah).
  • Priority: Seberapa *cepat* harus diperbaiki? (Lihat di bawah).
  • Environment: Di mana bug terjadi (misal: Android 13, v1.0.0).
  • Lampiran: Screenshot atau video rekaman.

3. Kunci Laporan Bug: Severity vs. Priority

Bug yang dampaknya besar (Severity) belum tentu urgensinya tinggi (Priority), dan sebaliknya. Ini adalah dua hal yang berbeda.

Severity (Tingkat Keparahan)

Seberapa besar dampak bug terhadap fungsionalitas? (Ditentukan oleh Tester/QA)

  • Critical: Aplikasi *crash*, data rusak, fungsionalitas utama GAGAL total.
  • Major (High): Fungsionalitas utama tidak berfungsi semestinya.
  • Minor (Medium): Fungsionalitas non-utama bermasalah atau bug kosmetik yang mengganggu.
  • Low: Bug kosmetik kecil (salah ketik, warna sedikit beda).

Priority (Tingkat Urgensi)

Seberapa cepat bug ini harus diperbaiki? (Ditentukan oleh Manajer Proyek)

  • P1 - Urgent: Harus diperbaiki SEKARANG. Menghalangi rilis.
  • P2 - High: Harus diperbaiki secepatnya, masuk dalam *sprint* ini.
  • P3 - Medium: Bisa diperbaiki di siklus rilis berikutnya.
  • P4 - Low: Akan diperbaiki jika ada waktu luang.

4. Cara Menghindari Bug (Pencegahan)

Pahami Persyaratan (Requirement) dengan jelas.
Terapkan Code Review antar pengembang.
Lakukan Unit Testing di awal pengembangan.
Manfaatkan Pengujian Otomatis (Automation).
Buat Rencana Pengujian (Test Plan) yang komprehensif.
Jaga Kolaborasi Tim (Developer, QA, Produk).

Pengantar Unit Testing

1. Apa itu Unit Testing?

Unit Testing adalah pengujian yang berfokus pada unit terkecil dari kode yang bisa diisolasi, seperti sebuah *function*, *method*, atau *class*. Tujuannya adalah untuk memverifikasi bahwa setiap unit kode bekerja persis seperti yang diharapkan.

Dasar Piramida Pengujian

Unit test adalah level paling dasar, paling banyak jumlahnya, dan yang paling cepat serta murah untuk dijalankan. Mereka membentuk fondasi yang kokoh untuk *Integration Test* dan *End-to-End (E2E) Test*.

Analogi: Suku Cadang Mobil

Anggaplah Anda sedang merakit mobil. Unit testing adalah proses menguji setiap komponen (roda, baut, mesin) secara terpisah, *sebelum* Anda merakitnya menjadi satu mobil utuh.

2. Pola Inti: Arrange, Act, Assert (AAA)

Hampir semua unit test ditulis menggunakan pola 3 langkah sederhana ini.

1. Arrange (Atur)

Menyiapkan kondisi awal dan semua data yang diperlukan untuk tes. (Setup).

2. Act (Aksi)

Menjalankan/memanggil *function* atau *method* yang sedang diuji.

3. Assert (Pastikan)

Memverifikasi bahwa hasil dari "Aksi" sesuai dengan yang diharapkan (*Expected Result*).

3. Contoh Praktis (Pola AAA dengan Pytest)

Kode Aplikasi (shopping_cart.py)

class ShoppingCart:
    def __init__(self):
        self.items = []

    def add(self, item: str):
        self.items.append(item)

    def get_items(self) -> list:
        return self.items

Kode Test (test_shopping_cart.py)

def test_add_item_to_cart():
    # 1. Arrange (Atur)
    cart = ShoppingCart()
    item = "apple"

    # 2. Act (Aksi)
    cart.add(item)

    # 3. Assert (Pastikan)
    assert cart.get_items() == ["apple"]

Framework Populer

Setiap bahasa memiliki *framework*-nya sendiri. Yang paling populer adalah: JUnit 5 (untuk Java), Jest (untuk JavaScript/React), dan Pytest (untuk Python).

4. Mengapa Unit Testing Penting?

Mendeteksi Bug Lebih Awal: Menemukan masalah di level fungsi, sebelum digabung.
Menghemat Waktu & Biaya: Bug yang ditemukan di awal jauh lebih murah diperbaiki.
Mempermudah Refactoring: Memberi kepercayaan diri untuk mengubah/memperbaiki kode tanpa merusak fitur lain.
Dokumentasi yang "Hidup": Test case berfungsi sebagai contoh cara penggunaan fungsi tersebut.

Ringkasan Materi: API Testing

1. Apa itu API Testing?

API (Application Programming Interface) Testing adalah proses pengujian yang berfokus pada "lapisan tengah" atau "lapisan logika bisnis" dari sebuah aplikasi, tanpa melalui antarmuka pengguna (UI).

Analogi Sederhana: Restoran

  • UI (Frontend) adalah pelayan dan menu tempat memesan.
  • API (Backend) adalah dapur yang menerima pesanan (request), memasaknya, dan mengirimkan makanan (response).
  • API Testing adalah proses menguji apakah dapur (API) dapat memasak pesanan (request) dengan benar dan menghasilkan makanan (response) yang sesuai, tanpa perlu berbicara dengan pelayan (UI).

2. Anatomi Panggilan (Call) API

Client Request

Permintaan yang dikirim dari klien ke server.

  • POST GET PUT DEL
    Method: Aksi yang ingin dilakukan.
  • URL (Endpoint): Alamat server yang dituju.
  • Headers: Info meta (misal: Tipe Konten, Kunci Otentikasi).
  • Body: Data yang ingin dikirim (biasanya format JSON untuk POST/PUT).

Server Response

Balasan yang diberikan server setelah memproses request.

  • 200 OK 201 Created
    Status Code (Success): Permintaan berhasil.
  • 401 Unauthorized 404 Not Found
    Status Code (Client Error): Kesalahan dari sisi klien (misal: salah URL, tidak login).
  • 500 Internal Error
    Status Code (Server Error): Kesalahan terjadi di server.
  • Body (Response): Data yang diminta (biasanya format JSON).

3. Mengapa API Testing Penting?

  • Mendeteksi Bug Lebih Awal: Menemukan masalah di logika bisnis sebelum UI dibuat.
  • Meningkatkan Keandalan & Keamanan: Memastikan API aman dari akses tidak sah dan dapat menangani beban.
  • Mengukur Performa: Menguji seberapa cepat API merespons di bawah tekanan (load testing).
  • Mempercepat Pengembangan: Pengujian API sangat cepat dan mudah diotomatisasi.

4. Tools Populer

Postman (Paling Populer)

Sangat user-friendly, bagus untuk manajemen collection (kumpulan request), dan mudah untuk testing otomatis.

SoapUI

Sangat kuat untuk skenario enterprise yang kompleks. Sangat baik untuk API berbasis SOAP (XML) dan mendukung pengujian security & load tingkat lanjut.

Pengantar Cypress (End-to-End Testing)

1. Apa itu Cypress?

Cypress adalah framework pengujian End-to-End (E2E) modern yang dirancang untuk menguji aplikasi web (seperti yang dibuat dengan React, Vue, Angular, dll). Cypress berjalan langsung di dalam browser, membuatnya sangat cepat dan interaktif.

Posisi dalam Piramida Testing

Cypress berada di puncak piramida (End-to-End). Ia menguji aplikasi Anda secara keseluruhan, sama seperti cara pengguna asli berinteraksi dengannya (mengklik tombol, mengisi formulir, dll).

Setup Sederhana

Hanya butuh Node.js. Instalasi dilakukan dengan satu perintah:

npm install cypress --save-dev

2. Keunggulan Utama Cypress

Test Runner Interaktif

Melihat tes berjalan secara real-time di browser saat Anda menyimpan kode.

Time Travel Debugging

Cypress mengambil snapshot dari setiap langkah, memungkinkan untuk "mundur" dan melihat apa yang terjadi.

Automatic Waits

Tidak perlu `sleep()` atau `wait(1000)`. Cypress otomatis menunggu elemen muncul sebelum beraksi.

Arsitektur Modern

Berjalan di event loop yang sama dengan aplikasi, memberikan kontrol penuh dan kecepatan tinggi.

3. Perintah Dasar (Sintaks)

Perintah Cypress sangat intuitif dan mudah dibaca:

cy.visit('https://..._')

// Membuka halaman web

cy.get('#username')

// Mencari elemen berdasarkan selector (ID, class)

cy.contains('Login')

// Mencari elemen berdasarkan teks di dalamnya

cy.get('#btn-login').click()

// Mengklik elemen

cy.get('#username').type('user_test')

// Mengetik ke dalam input field

4. Studi Kasus: Test Case Login

Contoh file tes Cypress (cypress/e2e/login.cy.js) untuk situs `saucedemo.com`:

describe('Login Test saucedemo.com', () => {

  beforeEach(() => {
    cy.visit('https://www.saucedemo.com/')
  })

  it('TC01: Login dengan kredensial valid', () => {
    cy.get('#user-name').type('standard_user')
    cy.get('#password').type('secret_sauce')
    cy.get('#login-button').click()
    cy.url().should('include', '/inventory.html')
  })

  it('TC02: Gagal login dengan password salah', () => {
    cy.get('#user-name').type('standard_user')
    cy.get('#password').type('wrong_pass')
    cy.get('#login-button').click()
    cy.contains('do not match').should('be.visible')
  })
})
